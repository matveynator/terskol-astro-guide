# chicha-astro-control wiki (RU)

## Назначение
`chicha-astro-control` — desktop-приложение на Go + WebView для мониторинга дискретных входов DI и управления дискретными выходами DO на платформах Vecow-класса.

## Что делает приложение
- Показывает 8 входов `DI1..DI8` в реальном времени.
- Управляет 8 выходами `DO11..DO18`:
  - ON/OFF;
  - software PWM (`0..100`).
- Сохраняет пользовательские подписи каналов и состояние выходов в JSON-конфиг.
- Открывает ссылку репозитория во внешнем браузере.


## Сборка
### Linux / macOS
```bash
go build -o /usr/local/bin/chicha-astro-control chicha-astro-control.go
chmod +x /usr/local/bin/chicha-astro-control
chicha-astro-control
```

### Windows
```bash
GOOS=windows GOARCH=amd64 go build -ldflags="-H windowsgui" -o chicha-astro-control.exe chicha-astro-control.go
```

## Параметры запуска
- `-DI` — шаблон пути к DI (`%d` обязателен)
- `-DO` — шаблон пути к DO (`%d` обязателен)
- `-config` — путь к JSON-файлу настроек

> На Windows для ECX-1000-2G штатный путь — через `ECX1K.dll` API.
> Флаги `-DI/-DO` используются как fallback только для Unix-like backend.

## Значения по умолчанию
- Linux: `/sys/class/gpio/gpio%d/value`
- Windows: `ECX1K.dll` (`Initial`, `SetGPIOConfig`, `GetGPIO`, `SetGPIO`)
- macOS: `/tmp/astro-control/di%d.value` и `/tmp/astro-control/do%d.value`

## GPIO backends
- Windows (`GOOS=windows`):
  - загрузка `ECX1K.dll`;
  - инициализация `Initial(0, 0)` для non-isolated GPIO;
  - настройка направлений `SetGPIOConfig(0xFF00)` (DI: bits `0..7`, DO: bits `8..15`);
  - чтение входов через `GetGPIO`, управление выходами через `SetGPIO`.
- Linux/macOS:
  - backend через файловые пути (`/sys/class/gpio` для Linux, `/tmp/astro-control` для macOS);
  - если пути недоступны — runtime автоматически переходит в simulation mode.

## Быстрая памятка по пинам
- `DI1..DI8` → пины `1..8`
- `DI_COM` → пин `9`
- `DIO_GND` → пины `10` и `19`
- `DO11..DO18` → пины `11..18`
- `External VDC` → пин `20`

## Таймер отключения.
В интерфейсе у каждого DO-канала есть таймер:
- при включении канала (`ON`) запускается обратный отсчёт;
- после истечения времени приложение автоматически отправляет обратную команду (`OFF`);
- если канал выключить вручную раньше, отсчёт для этого канала очищается.

### Как пользоваться таймером пошагово
1. Выберите нужный DO-канал.
2. Нажмите кнопку редактирования таймера (`✎`) рядом с полем времени.
3. Введите длительность и сохраните (`💾`).
4. Нажмите `ON` для запуска канала.
5. Дождитесь завершения отсчёта — канал автоматически вернётся в противоположное состояние.

### Поддерживаемые форматы длительности
- `ns`, `ms`, `s`, `m`, `h`, `d`, `M`, `y`
- Примеры: `500ms`, `0.5s`, `15s`, `2m`, `1h`
- Для `m/h/d/M/y` значение округляется вверх до целой секунды.
- Пустое значение интерпретируется как `0s` (watchdog фактически выключен).

### Практические рекомендации
- Для реле и силовых цепей начинайте с коротких интервалов (`1s..5s`) на тестовом стенде.
- Для аварийной логики используйте таймер как защиту от «залипания» канала в `ON`.
- Проверяйте, что у нагрузки корректно подключён общий `GND` (пины `19/10`) и внешнее питание `VDC` (пин `20`).

## План развития: автогидирование по звёздам

Ниже — практическая схема, как перейти от текущего режима `ON/OFF` к устойчивому сопровождению звезды (2–5 минут и далее) для обсерватории в районе Терскола.

### 1) Целевая архитектура (простая и расширяемая)
- **Подсистема кадра**: принимает FITS/RAW/JPEG от гида или основной камеры, нормализует яркость и шум.
- **Подсистема астрометрии**: определяет WCS (решение plate solve) и связывает пиксели с координатами неба.
- **Подсистема гидирования**: считает смещение целевой звезды относительно центра кадра и формирует коррекцию.
- **Подсистема монтировки**: переводит коррекцию в команды по осям (обычно RA/DEC, иногда Alt/Az + ротатор).
- **Подсистема купола**: отдельный контур, который держит щель купола синхронно с направлением телескопа.
- **Операторский UI**: экран гидирования с выбором звезды, кроссхером, графиком ошибки и логом команд.

### 2) Базовый контур управления (MVP)
1. Захватить кадр.
2. Найти звёзды и выбрать опорную (или несколько).
3. Посчитать центроид(ы) в пикселях.
4. Определить смещение `(dx, dy)` от целевой позиции.
5. Через калибровочную матрицу перевести смещение в команды монтировке.
6. Отправить импульс/скоростную коррекцию.
7. Повторять цикл 1–5 Гц.

На первом этапе этого достаточно, чтобы выйти на 2 минуты без заметного «плытия» при нормальном seeing и корректной полярке.

### 3) Алгоритмы, которые реально работают в полях
- **Сопоставление кадров**: фазовая корреляция (FFT) для грубого сдвига + центроид ярких звёзд для точной подстройки.
- **Точечное сопровождение**: weighted centroid или 2D Gaussian fit.
- **Робастность**: sigma-clipping по набору звёзд (игнорировать «плохие» звёзды/облака/шум).
- **Фильтрация**: PI/PID с anti-windup и ограничением шага коррекции.
- **Защита от перерегулирования**: deadband в пикселях + минимальный интервал между импульсами.

### 4) Каталоги звёзд и астрометрия
Для отображения «что за звезда» и быстрой валидации поля:
- **Gaia DR3** — основной точный каталог.
- **Tycho-2** — лёгкий справочный каталог для ярких звёзд.
- **UCAC4 / APASS** — дополнительные опорные источники в зависимости от FOV.
- **SIMBAD/Vizier** — для карточки объекта по имени и альтернативным идентификаторам.

Рекомендуемый подход: локальный кэш индексов для plate solving + онлайн-метаданные по запросу.

### 5) Что добавить в UI «экран гидирования»
- Live-кадр с центром и выделенной целевой звездой.
- Кнопки: `Select Star`, `Calibrate`, `Start Guiding`, `Pause`, `Dither`.
- Параметры: экспозиция, gain, частота цикла, aggressiveness RA/DEC, min-move.
- Графики: ошибка X/Y, суммарная RMS, длительности импульсов.
- Индикаторы: статус астрометрии, качество сигнала (SNR/HFD), saturation warning.

### 6) Калибровка и ввод в эксплуатацию
- **Нулевая фаза**: проверить масштаб (arcsec/pixel), ориентацию осей и задержку ответа монтировки.
- **Калибровка осей**: отдельные тест-импульсы по RA/DEC и измерение фактического сдвига.
- **Компенсации**: backlash, periodic error (если поддерживается), refraction (по необходимости).
- **Критерий приемки**: удержание звезды в пределах заданного радиуса не менее 120 секунд; цель — 300 секунд.

### 7) Купол (отдельный контур)
Для 4 асинхронных моторов купол лучше вести независимо от гидирования:
- вход: текущее направление телескопа + позиция купола;
- цель: держать щель с упреждением по азимуту;
- управление: плавный профиль скорости, межблокировки и контроль конечников;
- логика fail-safe: потеря телеметрии → безопасная остановка.

### 8) Минимальный технический roadmap
1. **v0.1**: загрузка кадров, ручной выбор звезды, расчёт `dx/dy`.
2. **v0.2**: калибровка матрицы «пиксели → импульсы осей».
3. **v0.3**: замкнутый контур гидирования и графики ошибок.
4. **v0.4**: интеграция plate solve + карточка звезды из каталога.
5. **v0.5**: раздельный планировщик купола и журнал диагностики.


### Первый контрольный milestone: офлайн-ошибка наведения

Перед подключением монтировки необходимо подтвердить, что система стабильно измеряет ошибку наведения по кадрам.

- Целевая метрика: не «звезда в центре гид-кадра», а измеряемый offset guiding относительно заранее откалиброванного офсета гид-камеры.
- Базовый тест: два кадра одного поля (`frame_001.fits` и `frame_002.fits`) и отчёт `dx`, `dy`, `rotation`, `confidence`.
- Минимальная робастность: использовать несколько звёзд (минимум 3–5), а не одиночный пик.
- До включения моторов: вывести условную коррекцию через матрицу `[[a,b],[c,d]]` и проверить качество решения на реальных кадрах.

Критерий готовности этапа:
1. чтение двух кадров одного поля;
2. автоматическое обнаружение и фильтрация звёзд;
3. сопоставление звёзд между кадрами;
4. оценка `dx`, `dy`, `rotation`, `confidence`;
5. отчёт для оператора с рекомендуемой условной коррекцией.

### Реализация шага 2: офлайн-анализ серии кадров

В `pkg/guiding` добавлен режим анализа серии кадров, где каждый кадр сравнивается с опорным первым кадром, а результат формируется как временной ряд ошибок наведения:

- `frame_index`
- `delta_x`, `delta_y`
- `rotation_deg`
- `confidence`
- `matched_stars`
- `suggested_motor_x_ms`, `suggested_motor_y_ms`

Это позволяет строить графики ошибки X/Y во времени и увидеть срыв сопровождения по кадрам до подключения реальных моторов.

### Реализация шага 3: live-кадры без управления моторами

Добавлены API для live-проверки гид-цепочки без замкнутого контура управления:

- `POST /api/guiding/live/start` — старт сессии и фиксация опорного кадра;
- `POST /api/guiding/live/frame` — анализ очередного live-кадра относительно опорного;
- `GET /api/guiding/live/state` — текущее состояние сессии (`processed/success/failed`, последний результат).

Сессия хранит только диагностику изображения: найденные совпадения, `dx/dy`, `rotation`, `confidence`. Команды на монтировку не отправляются.

### Реализация шага 4: ручные подсказки оператору

В live-снимке (`/api/guiding/live/state` и ответ `/api/guiding/live/frame`) добавлено поле `operator_hint`:

- `should_act` — надо ли оператору давать коррекцию;
- `axis_x_direction`, `axis_x_pulse_ms` — команда для ручного импульса по оси X;
- `axis_y_direction`, `axis_y_pulse_ms` — команда для ручного импульса по оси Y;
- `summary` — готовая текстовая инструкция вида: `Press +X for 100 ms and -Y for 40 ms.`

Подсказка учитывает deadband и минимальный импульс, чтобы не дёргать монтировку по шуму seeing.

### 9) Практические условия для Терскола
- Начинать с гид-экспозиций `1–2 с`, затем адаптивно увеличивать при слабом сигнале.
- Для ветра/турбулентности использовать меньше aggressiveness и больше усреднение.
- Вести полный лог: timestamp кадра, `dx/dy`, команда, ответ монтировки, температура/seeing.
- Сначала зафиксировать стабильные 2 минуты на нескольких объектах; после — оптимизировать до 5 минут.
